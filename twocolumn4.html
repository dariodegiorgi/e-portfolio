Evaluation Final project vs design proposal

Your evaluation of the final project (Unit 11) vs. the design proposal (Unit 6).

Evaluation Final project vs design proposal. During the design proposal several vulnerabilities were exposed, such as tampering, repudiation, information disclosure, denial of service, spoofing and elevation of privileges. During this phase of the work tools such as Ping, tracert, WHOis and Mxtoolbox were used to obtain basic information about the website and during the work we announced the tools that will potentially be used for the final project. Several hacking methods were highlighted in the design proposal that were not finally realized in the final project, such as following the dig command to find the public IP address with a denial-of-service attack could be realized. The display of domain names could allow poisoning of the DNS cache. The idea of the final project was more in the sense of highlighting solutions in order to limit attacks on the opposing website. During the implementation phase of the final project the tool used was more advanced and identified more vulnerabilities. The main problems encountered with regard to the security of the opposing website were the use of an insecure protocol such as http. The use of a wordpress CMS that could be exploited by hackers and various unsecured ports being open. There was also the version of the Linux kernel and javascript that was not up to date. The idea of the final project was to expose these issues by arguing with valid sources. By following the recommendations made during the final project, the vulnerabilities of the website can be considerably reduced. In conclusion, the final project allowed us to go deeper into the subject and to be more concrete about the security flaws, especially during the practical implementation and penetration tests. We can see that even a "simple" website has many security flaws.
